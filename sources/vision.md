InboxOS — Vision & Ideation

What This Website Actually Is

This isn't a marketing site for InboxOS — it's the engineering command center for the people building it. It's the single source of truth a contributor opens on Day 1 and never needs to leave: what to build, why, who owns what, and how it all fits together.

The Core Idea (InboxOS itself)

InboxOS is not an email client — it's a decision-and-execution layer that sits on top of Gmail/Outlook. The philosophy: Gmail is the hard drive, InboxOS is the operating system. It reads incoming email, decides what matters, and acts — so the user never has to open their inbox unless they choose to.

It works as a 5-layer pipeline: Ingestion → Intelligence → Decision → Action → Delivery. An email comes in, AI classifies and understands it, rules decide what to do, actions execute (create task, send alert, draft reply), and the outcome is delivered wherever the user lives — WhatsApp, Slack, dashboard.

What the Website Needs to Do (Purpose, Not Tech)


Onboard a contributor in under 5 minutes. Anyone landing here — beginner or professional — should immediately understand the vision, the architecture, and exactly where they fit.
Be the living roadmap. Not a static PDF — a navigable map of epics → features → tasks → PRs, so nobody has to ask "what's next" in a group chat.
Make ownership unambiguous. Every folder, every module, every task has a named owner and a backup. No confusion, no duplicate work, no merge conflicts from overlap.
Separate tracks by skill level, not by importance. Professionals own architecture-critical, high-risk modules. Beginners own fully isolated, well-scoped modules (100+ "Good First Issues," each under 2 hours, zero architecture risk) — so beginners feel real ownership without ever blocking or breaking the core system.
Show the full ambition, not just the MVP. The Feature Expansion page exists so contributors building the boring v1 plumbing can see where it's going — semantic search, a knowledge graph, a plugin marketplace, mobile/desktop apps, voice briefings. It's there to keep people motivated by the size of the vision.
De-risk the project openly. A dedicated Risk Analysis section means architecture, AI, security, and scaling risks are named upfront with real mitigations — not discovered mid-sprint.
Function as the open-source front door. Repo structure, contribution levels, RFC process, release cadence — everything a new open-source contributor needs to go from "interested" to "merged PR" without a single Discord message.


Who This Is For


The 13-person build team (5 professionals, 8 beginners) — this is their shared brain.
Future open-source contributors discovering the project and deciding whether to jump in.
You, as project lead — a way to think out loud, plan sprints, and keep 13 people moving in the same direction without constant re-explaining.


The Feeling It Should Give

Someone opening this site should feel: "This is a real, serious, well-thought-out engineering effort — and I can see exactly how I'd contribute to it today." Confidence and clarity over hype. It should read like internal documentation at a company that already knows what it's doing, not a hackathon pitch deck.

What It Deliberately Avoids


No sales language, no "join the waitlist," no consumer-facing polish for end users of InboxOS itself.
No ambiguity about ownership — every page ties back to "who does this" and "what does done look like."
No overwhelming wall of text on one page — the sidebar splits everything into Overview → Team → Planning → Issues & PRs → Strategy, so each page has one job.


One-Line Vision

A self-contained engineering HQ that turns "I want to build an AI email OS" into a 6-month, 13-person, fully-owned execution plan — readable by a beginner in 10 minutes and precise enough for a senior engineer to start shipping the same day.
